# i18n Spec

**Status**: Approved **Implementation**: `internal/i18n/` (engine), root package (public API) **Sources**: go-i18n (MIT) **Depends on**: Root package (`HTTPError`, `TranslationKeyer`), `validation/` (`Errors`, `ValidationError`) **External dep**: `golang.org/x/text/language` (quasi-stdlib, internal only) **ADRs**: [013-internationalization](../adr/013-internationalization.md)

---

## Overview

Credo's i18n system translates validation errors and HTTP errors into localized messages. The implementation lives in `internal/i18n/` (pure message lookup engine with zero root package imports). The public API consists of exactly 3 methods plus 2 types in the root package.

**Design principles**:

- Translation happens in the framework's **internal error handler**, not in the validation engine
- `internal/i18n` is a pure engine — no root package imports, no circular deps
- Key-based error translation via `TranslationKeyer` interface
- JSON-only locale files — zero external dependency for parsing
- Two-mode field name translation: field-agnostic (default) + field-aware (opt-in)

## API Surface

### I18nConfig (Setup Configuration)

```go
type I18nConfig struct {
    Dir     string              // filesystem path (e.g., "locales/")
    DirFS   fs.FS               // embed.FS or any fs.FS
    Default string              // default language (e.g., "en")
    Detect  func(*http.Request) string  // custom language detection
}
```

### app.UseI18n(cfg...) — Setup

```go
func (app *App) UseI18n(cfgs ...I18nConfig) error
```

UseI18n initializes i18n for the application:

1. No args or zero-value cfg → reads RawConfig `"i18n"` key → if absent, defaults: dir=`"locales/"`, default=`"en"`
2. Dir doesn't exist or is empty → returns nil (i18n inactive, no middleware)
3. Malformed files → returns error
4. Valid files → loads bundle, adds locale detection global middleware

### ctx.Locale() — Detected Language

```go
func (c *Context) Locale() string
```

Returns the detected locale string for this request (e.g., `"en"`, `"tr"`). Returns empty string if i18n is not configured.

### ctx.T(key, data...) — Translate

```go
func (c *Context) T(key string, data ...map[string]any) string
```

Translates a message key using the detected locale. Returns the key itself if i18n is inactive or the key is not found. Optional data map provides template variables.

### TranslationKeyer Interface

```go
type TranslationKeyer interface {
    TranslationKey() string
}
```

Implemented by error types that provide an i18n lookup key. `HTTPError` implements this: `TranslationKey()` returns `"http.404"`, `"http.500"`, etc.

## UseI18n Behavior

```go
// Zero-config — reads from RawConfig or uses defaults
app.UseI18n()

// Explicit config
app.UseI18n(credo.I18nConfig{
    Dir:     "locales/",
    Default: "en",
})

// Embedded filesystem
app.UseI18n(credo.I18nConfig{
    DirFS:   localeFS,
    Default: "en",
})

// Custom language detection
app.UseI18n(credo.I18nConfig{
    DirFS:   localeFS,
    Default: "en",
    Detect: func(r *http.Request) string {
        return r.URL.Query().Get("lang")
    },
})
```

When `Detect` is nil, defaults to reading the `Accept-Language` header.

## Key-Based Error Translation

The framework's internal error handler translates errors when i18n is active. Detection order:

1. `validation.Errors` → each error's `Code` mapped to `"v." + code` key
2. `TranslationKeyer` → uses `TranslationKey()` for key lookup, but only translates the message if the error is (or wraps) `*HTTPError`. Non-HTTPError TranslationKeyers are detected but returned unchanged.
3. `httpStatusProvider` → derives key from `HTTPStatus()` as `"http." + status`
4. Other errors → returned unchanged

The `translateError` function in root `errors.go` performs this dispatch using only `bundle.TranslateForLang()` and `bundle.FieldNameForLang()`. No `language.Tag` import needed in root.

## Two-Mode Field Name Translation

`ValidationError.Field` remains a **stable technical identifier** (JSON key) in all cases. Field name translation is **opt-in** via `fields.json`.

### Mode 1 — Field-Agnostic (Default)

Error messages do not reference field names. Messages like `"is required"` work standalone or are composed by the frontend.

### Mode 2 — Field-Aware (Opt-In)

When `fields.json` exists for the locale, the translated field name is injected into template data as `"field"`. Message templates can use `{{.field}}` to produce `"email address is required"`.

## Locale File Format

### Directory Structure

```
locales/
├── en/
│   ├── messages.json
│   └── fields.json      (opt-in)
└── tr/
    ├── messages.json
    └── fields.json
```

### messages.json

```json
{
  "v.required": "is required",
  "v.length": "must be between {{.min}} and {{.max}} characters",
  "http.404": "Not found",
  "http.500": "Internal server error"
}
```

### fields.json

```json
{
  "email": "email address",
  "first_name": "first name"
}
```

### Plural Support

```json
{
  "v.min_items": {
    "one": "must have at least {{.min}} item",
    "other": "must have at least {{.min}} items"
  }
}
```

Form selection is count-based and explicit: `ctx.TPlural(key, count, data...)` picks the CLDR cardinal form for the detected locale (delegated to `golang.org/x/text/feature/plural`) and exposes the count to the template as `{{.count}}`. `count` may be any integer kind, an integral float, or a decimal string (`"1.5"`) when visible fraction digits matter. `ctx.T` always renders the `other` form.

Like `ctx.T`'s key fallback, `ctx.TPlural` is tolerant: a count that cannot be interpreted as a number renders the `other` form instead of failing. The strict, error-returning path lives in the internal engine — `Localizer.LocalizePlural` rejects invalid counts (tolerant context helpers, strict engine).

## go-i18n Adaptation Scope

**Source**: [go-i18n](https://github.com/nicksnyder/go-i18n) (MIT) **Location**: `internal/i18n/` (not a public package)

### Adapted

| Feature | Credo use case |
| --- | --- |
| Bundle (message container) | Load locale files at startup, immutable after |
| Localizer (per-request lookup) | Internal message resolution |
| Message (ID + plural forms) | Locale file entries |
| Plural operand decomposition | Count → CLDR operands (form selection itself is delegated to `x/text/feature/plural`) |
| `text/template` substitution | `{{.min}}`, `{{.max}}`, `{{.field}}` params |
| BCP 47 matching (`x/text/language`) | Accept-Language resolution + fallback |

### Not Adapted

| Feature | Why skip |
| --- | --- |
| `Hash` field on Message | CLI merge workflow only |
| `goi18n extract/merge` CLI | Credo messages are programmatic |
| Custom delimiters | Unnecessary for error messages |
| `Must*` panicking variants | Errors are values |
| `RegisterUnmarshalFunc` | JSON-only format |
| Custom `TemplateParser` | `text/template` is sufficient |
| Generated CLDR rule tables (~1k lines) | Replaced by `x/text/feature/plural`; Unicode updates maintained upstream |

## Design Decisions

1. **Internal engine, root API** — `internal/i18n/` has zero root imports, preventing circular dependencies. Root exposes only `UseI18n`, `Locale()`, `T()`, `TPlural()`, `TranslationKeyer`.

2. **`TranslationKeyer` replaces `ErrorTranslator`** — Key-based lookup is simpler than type-dispatching interface. Error types declare their own translation key. No separate translator type needed.

3. **Translation in error handler, not validation engine** — `Rule[T].Validate()` stays stateless. Translation is a presentation concern.

4. **JSON-only locale files** — Zero external dependency, industry standard.

5. **Opt-in field name translation** — API-first defaults (Mode 1) + server-rendered opt-in (Mode 2 via `fields.json`).

6. **`golang.org/x/text/language` stays internal** — Never leaks into root package API. String-based APIs (`TranslateForLang`, `FieldNameForLang`) bridge the boundary.

7. **Tolerant helpers, strict engine** — `ctx.T`/`ctx.TPlural` never return errors: a missing key falls back to the key itself, an uninterpretable count falls back to the `other` form. Handlers should not branch on i18n failures. The internal `Localizer` is strict (errors are values) for callers that can handle them.
